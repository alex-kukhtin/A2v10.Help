Шаблони. Події

Шаблони

Об'єкт, що описує события в шаблоне, является звичайним об'єктом JavaScript. Імена властивостей цього об'єкта являются іменами подій, а значення - функціями, які будуть вызваны при виникненні события.

Імена подій конструируются различным образом в залежності від типа события:

  • для всей моделі імена фиксированные.
  • для события создания об'єкта конструируются з типа об'єкта і суффикса .constuct.
  • для подій изменения даних з полного пути к елементу даних і заданного суффикса.

Полный шлях до елементу даних фактически являє собою повне ім'я властивості об'єкта, начиная від кореня моделі. Для указания елементу масива використовується суффикс [].

Cобытия для всей моделі

Эти события имеют фиксированные імена.

  • Model.load - модель загружена.
  • Model.unload - модель выгружена. Никакие дії уже невозможны. Частіше за всі используюется для отписки от подій глобальной шины подій.
  • Model.saved - модель сохранена.
  • Model.beforeSave - перед зберіганням.

Аргументи для подій моделі:

  • this - корень моделі даних (IRoot).
  • model - корень моделі даних (IRoot).
  • caller - ссылка на корень моделі даних (IRoot) вызывающйого об'єкта. Например, якщо це модель діалога, то caller буде ссылкой на модель, з которой этот діалог був вызван. Может використовуватися для взаимодії с вызывающим елементом. Для подій Model.saved, Model.save значення завжди буде undefined.

Події для об'єктів

  • .construct - об`єкт создан. Префикс имени события являє собою ім'ятипа об'єкта. В обработчике цього события можна дополнить созанный елемент новыми властивостями. Однако для цього лучше використовувати механизм властивостей (properites).
  • .change - значення властивості изменено. Префикс имени события являє собою повний шлях до властивості в моделі даних.
  • .changing - значення властивості буде изменено. Префикс имени события являє собою повний шлях до властивості в моделі даних. Є можливість отменить изменения.

Аргументи для события .construct:

  • this - корень моделі даних (IRoot).
  • elem - созданий елемент.
  • prop - ім'я властивості батьківського елементу, которому буде присвоен созданий об'єкт. Имеет смысл для обработчиков создания елементів в MapObject.
Аргументи для подій .changing, .change:
  • this - корень моделі даних (IRoot).
  • elem - елемент.
  • newValue - новое значення властивості.
  • oldValue - старое значення властивості.
  • prop - ім'язменяемого властивості.
Возвращаемое значення має смысл тільки для события .changing:
  • Boolean - строгое значення false. Отменяет попытку изменения значення властивості. Любое другое значення разрешает изменения.
Зверніть увагу! При изменении значений об'єктных властивостей, оба значення властивості (и старое і новое) будуть указывать на один і тот-же об'єкт. Это происходит потому, что об`єкт никогда не пресоздается. Просто перезаписываются його властивості.

Події для масивів

Префикс имени для всех подій масивів являє собою повний шлях до властивості в моделі.

  • [].adding - перед добавлением елементу в масив.
  • [].add - елемент добавлен в масив.
  • [].change - изменен елемент масива.
  • [].remove - елемент удален з масива.
  • [].select - елемент масива выбран.

Зверніть увагу! Cобытия будуть генерироваться тільки тогда, когда для манипуляции с масивами використовуються средства платформи. Якщо масив меняется стандартними методами (наприклад push, splice), то события генерироваться не будуть.

Аргументи для подій масивів:

  • this - корень моделі даних (IRoot).
  • arr - изменяемый масив.
  • elem - новий або удаленный елемент.
Возвращаемое значення має смысл тільки для события .adding.
  • Boolean - строгое значення false. Отменяет операцию добавления елементу в масив.

Пользовательские события

Описанные выше события являются встроенными. Однако, никто не запрещает використовувати собственные события, які можуть более точно описывать бизнес-логику програми.

Такие события генерируются методом $emit кореневого об'єкта (IRoot) і можуть иметь произвольное ім'я(конечно не пересекающееся с системными іменами). Методом $emit також можна генерировать и системные события.

Зверніть увагу! Модель подій в системе связана с данными, а не с елементуми пользовательского інтерфейса. Другими словами, события возникают тогда, когда меняются дані. Возникновение подій никак не связано с пользовательским інтерфейсом. Фактично, пользовательского інтерфейса вообще може не быть.
